# CosS v0.5.3 全角色协作闭环测试报告

测试时间：2026-06-28  
测试方式：使用 Computer Use 真实操作 CosS 窗口，不仅看代码推测。  
测试任务：开发一个展示耐火材料研究室的主页，需要高级感。请产品经理、前端工程师、后端工程师、测试工程师、技术负责人和 DevOps 工程师分工协作完成闭环。

## 结论

本轮全角色协作闭环未完全通过。

已通过的部分：
- 任务拆解可生成 6 个角色子任务。
- 确认分派后可创建 6 个程序，并放入独立任务桌面。
- 任务视图可显示当前任务桌面为 6 个程序。
- 技术负责人、前端工程师、后端工程师的消息投递链路可走到 `投递 1/1`。
- 后端工程师和技术负责人 Agent 能进入 Claude Code 执行阶段，并出现真实文件创建确认。

未通过的部分：
- 多个系统模型子任务点击“开始执行”后会立刻变为“已阻塞”。
- 前端终端收到投递后只显示 `[Pasted text #1 +15 lines]`，未看到稳定执行反馈。
- Claude Code 文件写入确认需要人工审批，导致 Agent 无法无人值守完成闭环。
- 测试工程师、DevOps 的卡片操作在滚动布局下定位不稳定，未完成可靠投递闭环。
- 消息中心仍混入旧任务错误和旧任务消息，当前任务观察噪声较大。

## 实测流程

1. 打开已有 CosS v0.5.3 窗口，确认当前项目为 `D:\AgentTest`。
2. 使用任务视图确认当前任务桌面存在，并显示 `6 个程序`。
3. 从技术负责人开始，点击任务卡片的“开始执行”和“发送给角色”。
4. 在“发送给角色”弹窗中确认目标角色、任务频道和“同时注入运行中 Agent 终端”开关。
5. 在协作时间线中点击“确认投递”，观察投递状态。
6. 按相同方式测试前端工程师、后端工程师，并尝试滚动到测试工程师、DevOps。
7. 再次打开任务视图，复核桌面分组与程序数量。

## 关键缺陷

### P0：任务执行状态机异常

现象：产品经理、技术负责人、前端工程师点击“开始执行”后，部分任务立即从待执行变为已阻塞。  
影响：用户无法判断任务是真阻塞、模型失败、还是 UI 状态更新错误。全角色闭环会被错误中断。

建议：
- 记录 `startTask` 的模型调用结果、异常、状态迁移原因。
- 将“模型任务失败”和“人工标记阻塞”区分为不同状态。
- 不允许空响应、占位响应直接把任务置为已阻塞。

### P0：Agent 执行需要人工审批，闭环无法无人值守

现象：技术负责人、后端工程师收到任务后进入 Claude Code 文件创建确认，例如 `technical-architecture.md`、`ResearchProject.js`。  
影响：如果用户不手动批准，Agent 不能继续创建文件和完成任务。

建议：
- 在 CosS 中显式展示“等待 Claude Code 人工确认”状态。
- 支持配置 Agent 运行模式，例如只读、每次确认、允许本会话编辑。
- 不要把该状态误判为任务失败或已阻塞。

### P1：任务卡片滚动和点击定位不稳定

现象：在多窗口布局下，任务卡片滚动后，同一屏幕位置可能落到不同任务卡片上；测试过程中出现误触产品经理卡片状态的情况。  
影响：用户容易误操作任务状态，测试工程师和 DevOps 投递难以稳定完成。

建议：
- 为每个任务卡固定操作区，滚动后按钮不要漂移到难以判断的位置。
- 点击“开始执行”“标记完成”“标记阻塞”前增加轻量确认或撤销。
- 任务窗口支持放大、独立置顶和按角色过滤。

### P1：消息中心当前任务过滤不足

现象：消息中心会混入旧任务的 `COSS_AGENT_EVENT JSON 解析失败` 和旧任务状态消息。  
影响：当前任务协作状态被噪声淹没，用户难以判断本次任务是否正常。

建议：
- 默认只显示当前任务消息。
- 历史错误进入“历史/全部”过滤器。
- Agent 事件区显示任务名、时间和错误来源，并支持按严重级别过滤。

### P1：终端注入反馈不一致

现象：技术负责人、后端工程师收到指令后开始执行；前端工程师终端只显示 `[Pasted text #1 +15 lines]`，未见稳定响应。  
影响：用户无法确认指令是否真正提交给 Agent。

建议：
- 投递后检测终端是否出现新输出、是否停留在输入区、是否等待确认。
- 协作时间线增加投递后状态：已粘贴、已提交、Agent 已响应、等待人工确认。

## 通过项

- `v0.5.3` 版本号显示正常。
- 任务视图可显示独立桌面和程序数量。
- 角色窗口布局支持多窗口同时观察。
- “发送给角色”弹窗能正确带出目标角色。
- “同时注入运行中 Agent 终端”开关默认可用。
- 消息中心能记录 `task-instruction`、投递数量、`sent` 状态和 Agent structured event。

## 下一步建议

1. 优先修复任务状态机，确保“开始执行”不会无原因变成已阻塞。
2. 增加 Agent 等待人工确认的状态识别。
3. 优化任务窗口滚动、角色过滤和操作按钮定位。
4. 消息中心默认按当前任务过滤。
5. 增加投递后终端执行检测，区分粘贴成功和真正执行。
6. 修复后只跑本模块相关 e2e：任务状态机、消息投递、任务视图、多窗口布局和 Agent 终端注入。
