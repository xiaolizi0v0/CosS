# CosS 程序开发角色设计

版本：v0.1  
日期：2026-06-25  
适用范围：CosS 类桌面 AI 协作工作区的角色模板、任务分派和程序创建

## 1. 文档目标

本文档定义 CosS 中与程序开发相关的默认角色。角色是用户在类桌面工作区中看到和管理的核心对象。用户创建程序时，不应只选择“终端”“浏览器”“文件编辑器”，还应选择该程序代表什么角色，例如“前端工程师终端”“后端工程师终端”“测试工程师浏览器”。

这些角色用于：

- 用户右键桌面创建角色程序。
- 任务规划器把任务拆分给合适角色。
- 角色之间通讯、协作和交接。
- 设置不同角色的工具权限、命令权限和上下文范围。
- 在程序右下角圆形状态图标中展示协作关系。

## 2. 角色设计原则

### 2.1 角色面向职责，不面向工具

用户看到的是“前端工程师”“后端工程师”“测试工程师”，而不是“终端 1”“浏览器 2”。程序只是角色执行任务的载体。

### 2.2 一个角色可以使用多个程序

例如前端工程师可以同时拥有：

- 前端工程师终端
- 前端预览浏览器
- 前端代码编辑器
- 前端任务面板

### 2.3 一个程序类型可以创建多个角色

例如终端程序可以创建为：

- 前端工程师终端
- 后端工程师终端
- DevOps 工程师终端
- 测试工程师终端
- 安全工程师终端

凡是需要执行命令、修改代码、运行测试、使用 Claude Code 的角色，都可以创建 Claude Code 终端。

### 2.4 角色必须有权限边界

不同角色不应默认拥有相同权限。比如前端工程师通常不需要修改数据库迁移文件，测试工程师可以运行测试但不一定能直接提交发布，安全工程师可以读取敏感配置但写入操作需要确认。

### 2.5 角色协作应可见

每个角色程序右下角的圆形状态图标应显示：

- 该角色正在与哪些角色沟通。
- 该角色属于哪个任务组。
- 该角色当前是执行中、等待中、阻塞中还是需要用户确认。
- 最近协作对象和消息数量。

## 3. 角色模板字段

建议每个角色模板使用统一结构，方便任务调度器和 UI 调用。

```ts
type DevelopmentRoleTemplate = {
  id: string;
  name: string;
  category: "planning" | "implementation" | "quality" | "security" | "infra" | "data" | "design" | "release" | "documentation";
  description: string;
  responsibilities: string[];
  defaultPrograms: Array<"terminal" | "browser" | "editor" | "task_panel" | "log_viewer" | "api_client" | "database_client">;
  requiresClaudeCodeTerminal: boolean;
  recommendedTools: string[];
  fileScopes: string[];
  commandPermissions: "none" | "read_only" | "test_only" | "project_write" | "approval_required";
  networkPermission: "none" | "allowlist" | "full";
  commonCollaborators: string[];
  typicalTasks: string[];
  handoffOutputs: string[];
};
```

## 5. 详细角色定义

### 5.1 产品经理

角色定位：把用户需求转化为可开发、可验收的任务。

主要职责：

- 理解用户目标。
- 编写需求说明。
- 定义验收标准。
- 拆分功能范围。
- 判断哪些需求应进入当前版本。

默认程序：

- 任务面板
- 文档编辑器
- 浏览器

是否需要 Claude Code 终端：否，通常不需要。

推荐工具：

- 文档编辑工具
- 项目任务系统
- 原型图查看器
- 浏览器

常见协作对象：

- 系统架构师
- 前端工程师
- 后端工程师
- 测试工程师
- UI/UX 设计师

典型任务：

- 根据用户描述生成 PRD。
- 为功能生成用户故事。
- 编写验收标准。
- 判断任务优先级。

交付物：

- 需求文档
- 用户故事
- 验收标准
- 功能范围说明

### 5.2 项目经理

角色定位：管理任务节奏、依赖关系和交付状态。

主要职责：

- 维护任务列表。
- 跟踪阻塞问题。
- 协调角色之间的交接。
- 生成进度报告。
- 判断当前版本是否可交付。

默认程序：

- 任务面板
- 日志查看器
- 文档编辑器

是否需要 Claude Code 终端：否。

推荐工具：

- 任务管理工具
- 日志汇总工具
- 报告生成工具

常见协作对象：

- 产品经理
- 技术负责人
- 测试工程师
- 发布经理

典型任务：

- 汇总当前任务状态。
- 找出阻塞的角色和任务。
- 生成日报或版本报告。

交付物：

- 任务进度报告
- 阻塞清单
- 版本状态说明

## 9. 角色状态图标建议

|---|---|---|

## 12. 示例：创建“前端工程师终端”

用户操作：

1. 右键桌面。
2. 选择“新建终端”。
3. 在角色选择器中选择“前端工程师”。
4. 系统创建“前端工程师终端”窗口。
5. 终端启动 Claude Code，并注入前端工程师角色提示词。

系统行为：

- 设置工作目录为项目根目录或前端子目录。
- 限制默认写入范围为前端相关目录。
- 允许运行 npm/pnpm/yarn、测试和构建命令。
- 默认协作对象包括后端工程师、UI/UX 设计师、测试工程师。
- 在角色状态图标中显示当前执行和协作状态。

## 13. 示例：创建任务后的角色协作

用户创建任务：

```text
实现用户登录页面，并接入后端登录接口。
```

任务规划器拆解：

1. 产品经理确认验收标准。
2. 后端工程师检查或实现登录接口。
3. 前端工程师实现登录页面。
4. 测试工程师验证登录成功、失败、空表单、网络错误。
5. 技术负责人审查实现是否符合项目规范。

协作关系：

- 前端工程师与后端工程师确认接口字段。
- 前端工程师与 UI/UX 设计师确认页面状态。
- 测试工程师与前端工程师确认 bug 复现。
- 技术负责人对前端和后端输出审查意见。

状态图标表现：

- 前端工程师窗口右下角显示正在与后端工程师、测试工程师协作。
- 后端工程师窗口显示等待前端联调。
- 测试工程师窗口显示等待功能完成。
- 技术负责人窗口显示待审查。

## 14. 角色模板落地建议

建议将角色模板保存为可配置文件，例如：

```text
roles/
  product-manager.json
  tech-lead.json
  frontend-engineer.json
  backend-engineer.json
  qa-engineer.json
  ai-agent-engineer.json
  devops-engineer.json
  technical-writer.json
```

每个项目可以：

- 使用系统默认角色。
- 复制默认角色并自定义。
- 禁用不需要的角色。
- 为角色绑定不同模型。
- 为角色配置不同权限。
- 为角色设置默认程序和工作目录。

## 15. 总结

CosS 的角色系统应围绕“开发协作”来设计，而不是简单模拟一组聊天机器人。每个角色都需要有明确职责、工具权限、程序载体、协作对象和交付物。

MVP 阶段建议先实现 8 个默认角色，围绕项目创建、角色终端、任务拆解、角色通讯、协作角标形成闭环。后续再逐步扩展安全、性能、数据库、移动端、发布等专业角色。
