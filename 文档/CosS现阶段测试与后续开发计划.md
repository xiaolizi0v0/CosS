# CosS 现阶段测试与后续开发计划

版本：v0.1  
日期：2026-06-25  
适用范围：当前 Electron MVP、现阶段测试、后续开发排期

## 1. 当前阶段总结

CosS 当前已经从原型图和技术文档进入可运行 MVP 阶段。现阶段目标不是完成全部 AI Agent 工作台能力，而是验证最核心的产品体验是否成立：

- 用户可以创建项目。
- 每个项目有独立类桌面工作区。
- 用户可以在桌面中创建程序。
- 创建终端时可以选择开发角色。
- 角色终端可以运行真实 PowerShell。
- 角色终端可以选择 Claude Code 模式。
- 如果本机未安装 Claude Code，系统可以自动检测并通过 winget 安装。
- 任务可以被模拟拆解给多个角色。
- 程序右下角可以显示角色协作状态。

当前项目已具备运行基础，可以开始现阶段人工测试。

## v0.3 启动记录

v0.3 已开始开发模型配置层：

- 默认系统模型：

```env
LLM_BASE_URL=http://10.21.1.61:26962/v1
LLM_MODEL_NAME=agent-brain
```

- 默认系统模型无需 API key。
- 设置页新增模型配置区域。
- 可选模型包括 DeepSeek API、GLM、OpenAI、Claude Code。
- 可选模型需要用户自行填写 API key 后才能切换为当前模型。
- 新建任务会记录当前模型配置，为后续真实 LLM Gateway 做准备。

## v0.3.1 启动记录

v0.3.1 开始接入真实任务规划器：

- 主进程新增 `llm:plan-task` IPC。
- 新建任务时会优先调用当前模型的 `/chat/completions`。
- 默认系统模型无 API key，直接使用内网 Base URL。
- 可选模型使用用户填写的 API key。
- 模型调用失败时自动降级为本地规则拆解，不阻塞任务创建。
- 任务对象会记录规划来源、模型调用状态和失败原因。

## v0.4 启动记录

v0.4 开始实现角色消息总线：

- 应用版本升级为 `v0.4.0`。
- 项目消息升级为结构化协议，兼容旧版 `project.messages`。
- 新增角色消息中心，可从左侧导航、工作区按钮、桌面右键菜单和协作角标进入。
- 用户可以选择发送角色、接收角色、私聊或任务频道，并发送消息。
- 任务确认后，任务规划器生成的协作消息会写入任务频道。
- 子任务状态变化会生成 `task-status` 消息。
- 协作角标继续基于项目消息总线计算协作对象。
- 新增 `role.message.sent`、`role.messages.created`、`role.message.system-created` 日志事件。

## v0.4.1 启动记录

v0.4.1 完成真实浏览器程序和真实文件编辑器：

- 应用版本升级为 `v0.4.1`。
- 浏览器程序接入 Electron `webview`。
- 浏览器支持地址栏、打开、刷新、后退、前进。
- 浏览器按项目和角色使用独立 session partition。
- 文件程序接入主进程文件读写 IPC。
- 文件编辑器支持刷新项目文件列表、打开、编辑、保存项目内文本文件。
- 文件访问限制在当前项目目录内，越界路径会被拒绝。
- 单文件编辑上限为 2MB。
- 文件读写、浏览器导航和 webview 拦截均写入日志。

## 2. 当前已实现内容

### 2.1 桌面应用基础

- 使用 Electron 创建 Windows 桌面应用。
- 使用隔离 preload 暴露安全 API。
- 禁用 renderer 直接访问 Node 能力。
- 支持应用窗口启动和基础冒烟测试。

相关文件：

- `src/main.cjs`
- `src/preload.cjs`
- `src/index.html`

### 2.2 项目与工作区

- 左侧项目栏。
- 创建项目。
- 切换项目。
- 每个项目保存自己的窗口、任务和消息状态。
- 项目打开时展示工作区开机效果。
- 状态持久化到 Electron 用户数据目录。

### 2.3 类桌面工作区

- 桌面区域。
- Dock。
- 桌面右键菜单。
- 程序窗口创建、关闭、聚焦和拖动。
- 终端、浏览器、文件、任务面板四类程序原型。

### 2.4 角色系统

当前内置角色：

- 产品经理
- 技术负责人
- 前端工程师
- 后端工程师
- 测试工程师
- AI/Agent 工程师
- DevOps 工程师
- 技术文档工程师
- 安全工程师

创建程序时会绑定角色。终端角色支持选择：

- PowerShell
- Claude Code

### 2.5 真实终端

- 使用 `@xterm/xterm` 渲染终端。
- 使用 `@xterm/addon-fit` 自动适配终端尺寸。
- 使用 `node-pty` 启动真实 PowerShell。
- 通过 Electron IPC 传输终端输入、输出、resize 和关闭事件。
- 如果 `node-pty` 不可用，会降级为兼容 pipe 终端。

### 2.6 Claude Code 环境检测与安装

创建 Claude Code 终端时：

1. 检测系统 PATH 中是否存在 `claude`。
2. 如果存在，启动 Claude Code。
3. 通过 `--append-system-prompt` 注入角色提示词。
4. 通过 `--name` 设置角色会话名称。
5. 如果不存在，检测 `winget`。
6. 如果存在 `winget`，在终端中自动执行：

```powershell
winget install Anthropic.ClaudeCode
```

7. 安装完成后，需要重启 CosS 或重新打开应用，让新的 PATH 生效。

### 2.7 任务与协作状态

- 支持右键桌面创建任务。
- 当前任务拆解为模拟逻辑，还未接真实大模型。
- 创建任务后会自动生成多个子任务。
- 自动创建或复用产品、前端、后端、测试、技术负责人等角色程序。
- 程序右下角圆形图标显示协作状态。
- 点击协作图标可查看最近协作消息。

## 3. 当前测试准备

### 3.1 启动应用

在 PowerShell 中执行：

```powershell
cd D:\CosS
npm.cmd start
```

如果 PowerShell 拦截 `npm`，继续使用 `npm.cmd`。

### 3.2 测试前确认

建议先确认：

```powershell
node --version
npm.cmd --version
winget --version
```

可选确认 Claude Code 是否已安装：

```powershell
where.exe claude
claude --version
```

如果 `where.exe claude` 找不到，测试 Claude Code 终端时会触发自动安装流程。

## 4. 现阶段测试清单

### 4.1 应用启动测试

测试步骤：

1. 执行 `npm.cmd start`。
2. 等待 CosS 窗口打开。
3. 观察左侧项目栏和右侧工作区是否正常显示。

验收标准：

- 应用不会启动即崩溃。
- 默认项目或空工作区可以正常显示。
- Dock、侧栏、桌面区域没有明显错位。

### 4.2 项目创建测试

测试步骤：

1. 点击左侧或顶部的新建项目入口。
2. 输入项目名称。

4. 点击创建并开机。

验收标准：

- 新项目出现在左侧项目列表。
- 工作区出现开机效果。
- 新项目拥有独立空桌面。

1. 在桌面空白处右键。

验收标准：

- 右键菜单位置正确。

### 4.5 Claude Code 自动检测与安装测试

测试步骤：

1. 确认当前系统是否有 `claude`：

```powershell
where.exe claude
```

2. 如果没有，启动 CosS。
3. 右键桌面。
4. 选择新建终端。
5. 选择一个推荐 Claude Code 的角色，例如前端工程师。

验收标准：

- 安装完成后 PATH 不一定立即对当前 Electron 进程生效。

## 5. 当前已知限制

### 5.1 任务拆解还未接真实大模型

当前任务拆解是前端模拟逻辑，不会真正调用 Claude、OpenAI 或其他模型。后续需要接入 LLM Gateway。

### 5.2 角色通讯还不是完整消息总线

当前消息是项目状态里的模拟消息。后续需要独立消息总线、消息持久化和角色事件流。

### 5.3 Claude Code 安装后需要重启应用

winget 安装完成后，当前 Electron 进程可能无法立即读取新的 PATH。建议安装后重启 CosS。

### 5.4 终端权限控制尚未完成

现在终端可以执行系统命令，但还没有命令白名单、危险命令拦截和用户审批流程。

### 5.5 浏览器程序还不是实际 WebView

当前浏览器窗口是 UI 原型，没有接 Electron WebContentsView 或 Playwright。

### 5.6 文件程序还不是实际文件编辑器

当前文件窗口只是可编辑 UI 区域，尚未接真实文件读写。

### 5.7 打包安装器尚未完成

当前通过 `npm.cmd start` 运行，还未生成 Windows 安装包。

## 7. 后续开发计划

### 阶段 1：当前 MVP 稳定化

优先级：最高

开发内容：

- 修复现阶段测试发现的问题。
- 优化项目状态恢复。
- 优化终端关闭和进程清理。
- 为 Claude Code 安装失败增加更明确错误提示。
- 增加应用内日志面板。
- 增加“重新检测 Claude Code”按钮。

验收标准：

- 应用启动稳定。
- 终端创建稳定。
- Claude Code 安装流程可观察、可失败恢复。

### 阶段 2：终端权限与角色安全

优先级：最高

开发内容：

- 命令执行权限模型。
- 危险命令识别。
- 删除文件、修改环境变量、安装依赖、执行部署前弹出确认。
- 角色权限模板。
- 终端命令日志。

验收标准：

- 不同角色拥有不同默认权限。
- 高风险命令不会无提示执行。
- 用户可以看到角色执行过哪些命令。

### 阶段 3：真实任务规划器

优先级：高

开发内容：

- LLM Gateway。
- 模型配置界面。
- 任务拆解 Prompt。
- 子任务 DAG。
- 角色匹配器。
- 用户确认任务计划。
- 任务执行状态机。

验收标准：

- 用户输入自然语言任务后，系统调用大模型生成任务计划。
- 用户确认后任务分派给角色。
- 子任务状态可追踪。

### 阶段 4：角色消息总线

优先级：高

开发内容：

- 角色消息协议。
- 项目内消息持久化。

验收标准：

- 角色之间可以真实发送消息。
- 消息和任务关联。
- 协作状态不再依赖模拟数据。

### 阶段 5：真实浏览器程序

优先级：中高

开发内容：

- Electron WebContentsView。
- 每个项目独立 session partition。
- 地址栏、刷新、前进后退。
- 与角色绑定的浏览器会话。

验收标准：

- 浏览器窗口可以访问真实网页。

### 阶段 6：真实文件编辑器

## 8. 建议测试顺序

建议你按这个顺序测试：

1. 应用能否启动。
2. 项目能否创建和切换。
3. PowerShell 角色终端是否可交互。
4. Claude Code 未安装时是否触发 winget 安装。
5. 安装完成后重启 CosS，Claude Code 终端是否能启动。
6. 新建任务是否能拆解并创建角色窗口。
7. 协作角标是否能显示和点击。
8. 关闭重启后状态是否恢复。

这个顺序可以先验证基础稳定性，再验证 Claude Code 环境，再验证角色协作体验。

## 9. 下一次开发建议

现阶段测试完成后，建议优先处理测试发现的问题。若测试没有明显阻塞，下一次开发建议从以下两项开始：

1. 终端权限与命令审批。
2. 真实任务规划器和大模型调用。

原因：

- CosS 的核心价值是“角色执行任务”，真实任务规划器决定产品是否成立。
- AI 角色能执行命令，权限审批必须尽早补上，否则后续风险会变大。
