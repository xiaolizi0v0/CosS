## 1. 文档检测结论

旧版《CosS现阶段测试与后续开发计划.md》仍保留了早期 MVP 阶段判断，其中部分内容已过期：

- “任务拆解还未接真实大模型”已过期：v0.3.1 起已接入 LLM 任务规划，并支持失败降级。
- “角色通讯还不是完整消息总线”已过期：v0.4 起已实现角色消息中心和结构化消息协议。
- “浏览器程序还不是实际 WebView”已过期：v0.4.1 起已接入真实浏览器。
- “文件程序还不是实际文件编辑器”已过期：v0.4.1 起已接入项目文件读写。
- “Claude Code 深度集成尚未开始”已过期：v0.5.x 已进入 Claude Code、Codex、CodeBuddy 多 Agent 后端集成阶段。

当前阶段不再是基础 MVP 验证，而是进入“多 Agent 真实协作闭环稳定化”阶段。后续开发重点应从“能不能创建程序”转为“Agent 是否能稳定接收任务、持续执行、保留上下文、可观察、可恢复、可审计”。

## 2. 当前已完成能力

### 2.1 项目、对话、任务模型

- 项目工作区已可创建和切换。
- v0.5.5 已把“任务视图”调整为“对话视图”。
- 一个对话代表一个持续桌面。
- 用户可以在同一对话中连续发布多个任务，并复用已有角色程序。
- 任务列表窗口已新增，可按当前对话展示任务、模型、进度和子任务状态。

### 2.2 真实终端与终端留痕

- PowerShell、Claude Code、Codex、CodeBuddy Code 已进入统一 Agent 终端体系。
- 终端后端会话支持保活。
- 切换项目或切换对话时，不再直接杀掉真实终端进程。
- 回到原对话时可重新挂载 xterm 前端并回放 transcript。

### 2.3 角色消息与协作时间线

- 消息中心已从纵向列表升级为横向协作时间轴。
- 支持角色私聊、任务频道、分叉消息、节点详情、投递状态、终端输出引用。
- 投递状态已细化为 `submitted`、`responded`、`waiting` 等。
- 已修复时间轴滚动回到开头、节点断线、状态节点不共线、输入框样式不统一等问题。

### 2.4 Agent 执行状态

- 已新增 `waiting` 子任务状态，用于表示等待人工确认。
- Claude Code / Codex 审批等待提示可被识别。
- Agent 输出 `COSS_AGENT_STATUS:*` 与 `COSS_AGENT_EVENT:{...}` 可同步到任务和时间线。
- 已处理 `[Pasted text ...]`、状态示例文本等误识别问题。

### 2.5 文件和浏览器程序

- 文件程序已支持项目内文件树、打开、编辑、保存、新建文件夹、重命名、删除、另存为。
- 文件访问由主进程限制在当前项目目录内。
- 浏览器程序已支持真实网页访问、多标签、收藏、历史和任务 URL 打开。

### 2.6 Agent 后端设置

- 智能体设置已包含 Claude Code、Codex、CodeBuddy Code。
- 支持手动环境检测，不再进入设置页自动检测。
- 支持 Agent 登录测试。
- CodeBuddy Code 支持用户填写 API Key，并通过环境变量注入。
- CodeBuddy Code 真实交互投递已完成修复和人工验证。

## 3. 近期已解决问题

### 已解决：CodeBuddy Code 真实交互投递

历史现象：CodeBuddy TUI 中，CosS 已能把“读取投递文件”的指令输入到交互输入框，但真实提交曾出现未触发的问题。

解决结果：
- 已改为生成 `.coss/deliveries/*.md` 投递文件。
- 终端只输入一行读取文件指令，避免多行内容变成 `[Pasted text #1: n lines]`。
- 已改为分段输入、延迟提交，并追加增强 Enter 兜底。
- 已完成真实交互投递验证，CodeBuddy Code 可正常接收 CosS 投递并开始执行。

后续只需要把该能力保留为回归测试项，不再作为 P0 阻塞问题。

## 4. 现阶段剩余问题

### P0：多 Agent 后端投递协议需要统一

当前 Claude Code、Codex、CodeBuddy Code 的交互方式不完全一致：

- Claude Code 可较稳定处理 bracketed paste。
- Codex CLI 需要检测安装、登录态和命令路径。
- CodeBuddy Code 对粘贴占位、输入框提示和 Enter 提交更敏感。

后续需要抽象统一的 Agent 投递适配层：

- `prepareDelivery`
- `writeDelivery`
- `submitDelivery`
- `detectResponse`
- `detectWaitingApproval`
- `recoverStuckInput`

每个 Agent 后端只实现自己的适配器，消息中心和任务系统不再关心具体 CLI 差异。

### P0：Agent 权限与审批模式仍需产品化

v0.5.4 已识别“等待人工确认”，但还没有完整权限模式。

后续需要支持：

- 只读模式。
- 每次编辑都确认。
- 本会话允许编辑。
- 本会话允许安装依赖。
- 禁止部署、删除、格式化磁盘等高风险操作。
- 不同角色默认权限不同。

这是后续真实协作闭环的安全底座，应优先级很高。

### P1：任务列表仍是基础版

v0.5.5 已新增任务列表窗口，但仍缺少：

- 搜索任务。
- 按角色过滤。
- 按状态过滤。
- 任务归档。
- 任务拖动排序。
- 打开任务详情。
- 查看任务关联消息、投递和终端输出。

任务列表应成为“对话内任务管理中心”，而不仅是只读列表。

### P1：对话数据模型仍保留历史 `desktop` 字段

当前为了兼容历史数据，仍保留 `desktopId`。

后续需要分阶段迁移：

- UI 文案统一使用“对话”。
- 新数据写入 `conversationId`。
- 历史数据加载时自动迁移。
- 内部函数逐步从 `desktop` 命名迁移到 `conversation`。
- 保留一版兼容后，再删除旧字段依赖。

### P1：消息中心还需要更强的观察能力

当前横向时间线已可读，但真实多角色协作中仍需要：

- 按严重级别过滤。
- 按 Agent 后端过滤。
- 按投递状态过滤。
- 终端输出全文搜索。
- 按投递批次折叠。
- 将“系统事件、角色消息、Agent 输出、错误日志”分层显示。

### P1：真实浏览器需要补齐产品级能力

已完成真实浏览器基础能力，但还应补：

- `target=_blank`、新窗口链接统一拦截到 CosS 内部标签或当前浏览器程序。
- favicon 持久化。
- 标签拖拽排序。
- 跨角色共享收藏夹。
- 下载管理。
- 页面加载失败和证书错误提示。
- 测试工程师角色可基于浏览器执行页面验收。

### P1：文件编辑器需要从基础 textarea 升级

当前文件编辑器可用，但仍缺少专业编辑体验：

- 接入 Monaco Editor 或 CodeMirror。
- 语法高亮。
- 文件目录树展开/折叠。
- 文件差异预览。
- 外部修改冲突检测。
- 未保存变更提醒。
- Markdown 预览。
- 大文件和二进制文件保护。

### P2：数据存储仍应升级为 SQLite

当前状态仍以 JSON 状态文件为主。随着任务、消息、终端输出、日志和投递记录增长，JSON 会越来越脆弱。

后续需要：

- SQLite 数据库。
- 项目、对话、窗口、任务、消息、投递、Agent 事件、日志表。
- 数据迁移脚本。
- 导入导出。
- 异常恢复。

### P2：安装包与自动更新仍未完成

后续需要：

- Windows 安装包。
- 应用图标和开始菜单入口。
- 自动更新策略。
- 崩溃日志收集。
- 用户数据目录迁移策略。

## 5. 建议版本路线

### v0.5.6：Agent 投递稳定化

目标：在 CodeBuddy Code 真实投递已修复的基础上，统一 Claude Code、Codex、CodeBuddy Code 的投递适配层，让后续消息中心和任务系统不再关心具体 CLI 差异。

开发内容：

- 抽象 Agent 投递适配层。
- 保留 CodeBuddy Code 真实交互投递为回归测试项。
- 投递后检测是否仍停留在输入框。
- 卡住时支持重新提交、清空输入框、重新生成投递文件。
- 投递详情中显示投递文件路径、提交方式、最近终端输出。
- 增加真实手工测试清单。

验收标准：

- 三种 Agent 后端都可以收到同一条 CosS 指令。
- 时间线能显示“已提交、已响应、等待确认、已完成、已阻塞”。
- CodeBuddy Code 投递回归测试持续通过。

建议测试：

```powershell
node --check src\main.cjs
node --check src\renderer.js
node --check src\preload.cjs
node --check tests\e2e\app.spec.cjs
npm.cmd run test:e2e -- tests/e2e/app.spec.cjs -g "CodeBuddy|agent delivery|queues timeline messages"
```

并补充真实 CosS 窗口人工测试：

- Claude Code 投递。
- Codex 投递。
- CodeBuddy Code 投递。
- 投递后切换对话再切回，任务不断。

### v0.5.7：对话任务管理增强

目标：让“项目 -> 对话 -> 任务”成为稳定的信息架构。

开发内容：

- 任务列表搜索。
- 按角色、状态、模型过滤。
- 任务详情抽屉。
- 任务归档。
- 任务排序。
- 任务与消息、投递、终端输出关联查看。
- 数据字段从 `desktop` 命名逐步迁移到 `conversation`。

验收标准：

- 用户能在一个对话内连续发布多个任务并清楚管理。
- 任务列表可定位每个任务的角色、状态、消息、输出。
- 历史项目数据仍能正常加载。

### v0.6.0：权限与审批系统

目标：让 Agent 能执行真实任务，同时把风险控制在用户可理解、可审批、可追踪的范围内。

开发内容：

- Agent 权限模式：只读、每次确认、本会话允许编辑、本会话允许安装依赖。
- 按角色配置默认权限。
- 高风险命令识别下沉到主进程终端写入层。
- 文件写入、依赖安装、删除、部署、网络请求等动作进入审批流。
- 权限审批日志。

验收标准：

- 高风险操作不会无提示执行。
- 用户能看到 Agent 为什么等待确认。
- 用户能决定允许一次、允许本会话、拒绝。

### v0.6.1：浏览器和文件编辑器专业化

目标：让浏览器和文件程序从“能用”进入“可长期开发使用”。

开发内容：

- Monaco/CodeMirror 编辑器。
- 文件树展开/折叠、差异预览、外部冲突检测。
- 浏览器新窗口拦截、favicon、标签排序、下载管理。
- 测试工程师角色的浏览器验收入口。

验收标准：

- 用户能在 CosS 内完成基础代码查看、编辑、保存和页面验证。
- 浏览器链接不会跳出 CosS 主窗口造成工作流断裂。

### v0.7.0：数据存储与安装包

目标：让 CosS 具备更稳定的本地数据和可分发能力。

开发内容：

- SQLite 状态存储。
- JSON 到 SQLite 迁移。
- Windows 安装包。
- 自动更新基础能力。
- 崩溃日志和诊断包导出。

验收标准：

- 用户无需命令行启动。
- 大量消息和终端输出不会拖垮单个状态文件。
- 出问题时可以导出诊断资料。

## 6. 当前阶段推荐优先级

1. 先完成 v0.5.6：Agent 投递稳定化。
2. 再做 v0.5.7：对话任务管理增强。
3. 然后进入 v0.6.0：权限与审批系统。
4. 文件编辑器、浏览器增强可在 v0.6.1 做。
5. SQLite 和安装包放到 v0.7.0。

原因：

- CosS 的核心价值是多 Agent 协作执行任务，Agent 投递必须先稳定。
- 对话和任务管理决定用户能不能长期使用一个工作区。
- 权限系统越晚做，后续真实 Agent 能力越难安全落地。
- 文件、浏览器、存储、安装包很重要，但应排在协作闭环稳定之后。
